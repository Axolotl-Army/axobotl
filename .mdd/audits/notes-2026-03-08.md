# MDD Audit Notes — 2026-03-08

## [01] Leveling System

**Files read:** src/shared/models/UserLevel.ts, src/bot/utils/levelUtils.ts, src/bot/events/messageCreate.ts, src/bot/commands/rank.ts, src/bot/commands/leaderboard.ts, src/bot/commands/xp.ts, src/bot/commands/levelconfig.ts, tests/unit/leveling.test.ts

**Findings:**

- 🔴 HIGH — XP column is PostgreSQL `INTEGER` (max 2,147,483,647) but `computeXpUpdate` clamps at `Number.MAX_SAFE_INTEGER` (~9 quadrillion). Any user reaching ~2.1B XP will get an uncaught DB overflow error instead of being clamped. The `SET_MAX` constant (10,000,000) and natural gain rate make this unlikely but the model type and guard are mismatched.

- 🟡 MEDIUM — Cooldown is recorded (`recordXpAwarded`) before the DB write succeeds (`messageCreate.ts:24`). If `findOrCreate` or `record.update` fails, the user's 60-second cooldown window is burned but no XP is awarded.

- 🟡 MEDIUM — `leaderboard.ts` fetches each member individually inside `Promise.all` (up to 10 separate Discord REST calls). This can be slow and may hit rate limits. Discord's `guild.members.fetch({ user: [...ids] })` can batch fetch all at once.

- 🟡 MEDIUM — `levelconfig.ts` calls `Guild.update()` without verifying the guild row exists. If a guild has never been upserted into the `guilds` table, the update silently does nothing and the user sees a false success confirmation.

- 🟡 MEDIUM — `messageCreate.ts` does not check `message.system`. System messages (e.g. "User pinned a message", server boost notifications) fire `messageCreate` with a real user as author and `message.system = true`. These can award XP unintentionally.

- 🔵 LOW — `lastXpAt` is written to the DB on every XP award but never read. Cooldown state lives entirely in the in-memory `cooldownStore`. The column serves no runtime purpose and could mislead future developers. If in-memory cooldown is the intended approach (acceptable on restart), this column is dead weight; if persistent cooldown is desired, it should be wired up.

- 🔵 LOW — `/rank` always shows "Progress to Level N+1" even for level 0 users with 0 XP, displaying "0 / 50 XP". Functionally correct but slightly odd UX for users who have never chatted.

**Test coverage:** 35 unit tests covering levelUtils pure functions only. Zero tests for the four slash command handlers and the messageCreate event handler (these require Discord.js mocking and are not covered).

**Doc accuracy:** Good overall. One minor discrepancy: the doc still references the inverse formula `currentLevel = floor((totalXp / 50)^(1 / 1.7385))` but the implementation uses an iterative adjustment loop on top of that approximation (due to floating-point rounding). The description is accurate enough but could note the adjustment.
