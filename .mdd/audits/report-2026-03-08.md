## Findings

### HIGH

**H1 — XP column type vs guard mismatch** (`src/shared/models/UserLevel.ts`, `src/bot/utils/levelUtils.ts`)

The `xp` column is `INTEGER` in PostgreSQL (signed 32-bit, max ~2.1B). `computeXpUpdate` clamps at `Number.MAX_SAFE_INTEGER` (~9 quadrillion) — a value PostgreSQL will never accept. A user reaching ~2.1B XP would cause an unhandled `SequelizeDatabaseError` from the DB rather than a graceful cap.

**Fix:** Change the `xp` column to `BIGINT` in `UserLevel.ts`, or lower the cap in `computeXpUpdate` to `2_147_483_647`. BIGINT is the better choice long-term.

---

### MEDIUM

**M1 — Cooldown recorded before DB write** (`src/bot/events/messageCreate.ts:24`)

`recordXpAwarded()` is called before `findOrCreate` and `record.update`. If the DB write fails, the cooldown is consumed and the user loses their next XP award opportunity for 60 seconds.

**Fix:** Move `recordXpAwarded(guildId, userId)` to after the successful `record.update()` call.

---

**M2 — Leaderboard fetches members one-by-one** (`src/bot/commands/leaderboard.ts:31-41`)

Up to 10 individual `guild.members.fetch(userId)` calls are made in parallel. Discord's REST API is called once per user. This is slow and risks rate-limiting.

**Fix:** Use `guild.members.fetch({ user: topRecords.map(r => r.userId) })` to batch-fetch all members in a single call, then look up names from the returned Collection.

---

**M3 — `levelconfig` silently no-ops if guild row is missing** (`src/bot/commands/levelconfig.ts:33`)

`Guild.update()` silently does nothing if no row exists for the guild. The user receives a false success with preview, but nothing is saved.

**Fix:** Use `Guild.upsert()` instead of `Guild.update()`, or check the update's affected row count and surface an error if zero.

---

**M4 — System messages can earn XP** (`src/bot/events/messageCreate.ts:16`)

The `message.author.bot` check does not catch Discord system messages (pin notifications, boost messages, join messages). These have a real user as author with `message.system === true`.

**Fix:** Add `if (message.system) return;` after the bot check.

---

### LOW

**L1 — `lastXpAt` DB column is dead weight** (`src/shared/models/UserLevel.ts`)

The column is written on every XP award but never read. The cooldown runs entirely from in-memory state. Either wire it up (read `lastXpAt` on startup to restore cooldowns) or remove it to avoid confusion.

---

**L2 — `/rank` progress display at level 0** (`src/bot/commands/rank.ts`)

A user with 0 XP sees "Progress to Level 1: 0 / 50 XP". Functionally correct, but minor UX oddity.
